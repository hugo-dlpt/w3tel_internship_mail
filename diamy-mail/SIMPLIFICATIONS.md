rend la maquette **honnête** en revue. Règle d'or (guide §12) : si tu modifies la conception,
tu mets à jour l'annexe concernée (version + changelog) — jamais de divergence code/spec.

## Tests automatisés (`cargo test --workspace`)

Nécessitent un Postgres de dev actif (`docker compose up`) : `diamy-mxd` (6 tests, dialogue
SMTP réel contre un vrai serveur en écoute sur un port éphémère, dont un test STARTTLS
complet : négociation TLS réelle, session chiffrée, métadonnées TLS vérifiées) et
`diamy-maild` (3 tests, API de sync réelle par HTTPS). Discipline d'isolation contre la base
partagée (potentiellement concurrente entre binaires de test) : **jamais de `TRUNCATE`**
global ; chaque test génère son propre `device_id`/`message_id` frais et retrouve SON
message par un marqueur unique dans le contenu déchiffré plutôt que par comptage ou "le plus
récent" ; `aubin@w3.tel` est réservé comme principal "jamais enrôlé" dans toute la suite.
Vérifié stable sur plusieurs exécutions complètes consécutives de `cargo test --workspace`.

Le vérificateur de certificat "accepte tout" (`AcceptAnyCert`/`danger_accept_invalid_certs`)
n'est utilisé QUE côté client de test/exemple, jamais côté serveur — un client de test se
connecte à son propre serveur éphémère avec un certificat auto-signé qu'il ne peut pas
vérifier via une vraie chaîne de confiance ; ce n'est pas un contournement de sécurité côté
serveur (le serveur, lui, présente toujours un certificat et négocie toujours un vrai TLS).

## Garde-fous en place
- `dev-crypto` **refuse de démarrer hors `DIAMY_ENV=dev`** (`assert_backend_allowed_for_env`, A18 SEC-FC-1).
- Les données `dev-crypto` **ne sont pas interopérables** avec `messaging-crypto` → re-provision à la bascule, pas de migration.
- La démo `diamy-maild` **vérifie** (contre le vrai Postgres + l'object store local) qu'aucune ligne `mail.messages`/`mail.blobs` ne contient le clair d'origine (INV-1) ; le digest `mail.blobs.sha512_ct` est calculé sur le CHIFFRÉ, jamais le clair (A21-BLOB-1, A21-X-3).
- Le DDL de `crates/diamy-mail-storage/migrations/0001_mail_core.sql` est copié **verbatim** depuis A21 §2.1-2.4 (la DDL prévaut, A21 §1) ; `diamy-mxd` ET `diamy-maild` partagent la même crate `diamy-mail-storage`, donc le même catalogue Postgres — pas deux implémentations qui pourraient diverger (esprit de l'erreur IA n°1 d'A24 appliqué au stockage).
- `diamy-mxd` borne réellement la mémoire par ligne ET par message (A01-STAB-1) : une ligne SMTP sans retour à la ligne au-delà de 8 Ko est drainée sans être stockée, un corps DATA au-delà de 10 Mo est rejeté (`552`) — testé avec un message de 11 Mo, qui est bien rejeté et ne laisse aucune trace en base.
- Isolation des échecs par destinataire (A01-PIPE-3) : un destinataire inconnu est rejeté (`550`) sans bloquer les autres ; le SMTP `250` n'est renvoyé qu'après que la persistance Postgres a committé (A01-PIPE-1).
- Chemin vertical vérifié entre VRAIS processus séparés (pas une démo qui joue les deux rôles) : un mail envoyé par SMTP à `diamy-mxd` est lu et déchiffré par `read_test_mail`, un second processus qui ne parle qu'au réseau à `diamy-maild` et ne connaît que sa propre clé privée locale.
- TLS vérifié avec des clients INDÉPENDANTS de notre propre code de test : `openssl s_client -starttls smtp` négocie bien STARTTLS contre `diamy-mxd` (TLS 1.3, `TLS_AES_256_GCM_SHA384`, certificat signalé self-signed comme attendu) ; un client HTTPS Python indépendant négocie bien contre l'API de sync de `diamy-maild` (même suite). Les deux confirment le protocole réel, pas seulement notre propre harnais.
- L'API de sync rejette toute requête sans AppKey Tier 2 valide **avant** d'examiner le jeton mail-plane (A17-APPKEY-5) : testé avec une AppKey invalide + un jeton lui-même invalide → l'erreur observée est bien `ERR_APPKEY_INVALID`, jamais `ERR_TOKEN_INVALID` (si le jeton avait été vérifié en premier, l'erreur observée aurait été différente — c'est la preuve de l'ordre, pas une affirmation).
- `body_ct`/`summary_ct` sous `k_msg` portent désormais une AAD liée à leur `message_id`/`blob_id` exact (A02-CRY-2/3) : testé qu'un chiffré scellé sous l'AAD d'UN message échoue à s'ouvrir sous l'AAD d'un AUTRE (tag GCM rejeté, fail-closed).
- `RLIMIT_CORE` est réellement mis à 0 en prod (vérifié par `getrlimit` après appel, pas seulement "pas d'erreur retournée") et laissé INTACT en dev ; le démarrage échoue si la désactivation échoue en prod (fail-closed, A18-ZERO-4).
